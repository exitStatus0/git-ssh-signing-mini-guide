# Подпись Git-коммитов с помощью SSH-ключей

> Мини-курс и практическое руководство для YouTube-видео и GitHub-репозитория.

---

## Описание курса

Этот мини-курс объясняет, как подписывать Git-коммиты и теги SSH-ключом, как настроить автоматическую подпись коммитов в Git и как проверять эти подписи локально и на платформах вроде GitHub, GitLab и Bitbucket.

В отличие от классического сценария с GPG, подпись через SSH обычно проще настроить, потому что многие разработчики уже используют SSH-ключи для доступа к Git. GitHub прямо описывает SSH как самый простой вариант для большинства индивидуальных пользователей, а для проверки SSH-подписей Git требует версию **2.34 или новее**. GitHub, GitLab и Bitbucket могут показывать метку **Verified** для корректных коммитов, подписанных через SSH, если ключ подписи привязан к вашему аккаунту, а метаданные коммита соответствуют требованиям платформы.

---

## Почему это стоит делать

Подпись коммитов решает другую задачу, чем просто аутентификация при работе с Git по SSH.

Аутентификация отвечает на вопрос:

**«Может ли эта машина отправлять изменения в репозиторий?»**

Подпись коммита отвечает на другой вопрос:

**«Могу ли я криптографически доказать, кто создал этот коммит?»**

Это важно, потому что подпись коммитов:

- повышает доверие к истории изменений;
- помогает ревьюерам подтверждать происхождение коммитов;
- делает ваши изменения помеченными как **Verified** на платформах для хостинга репозиториев;
- снижает неоднозначность в командной работе, автоматизации и open-source-сотрудничестве;
- усложняет подмену истории и выдачу себя за другого автора;
- помогает организациям требовать подписанные коммиты в защищённых ветках.

Для аудитории YouTube главный посыл такой:

> SSH-подпись — это один из самых простых «маленьких апгрейдов безопасности», который можно добавить в ежедневный Git-процесс.

После этого руководства студент сможет:

## Предварительные требования

Перед началом убедитесь, что у вас есть:

- **Git 2.34+** для проверки SSH-подписей;
- SSH-ключ или право создать его;
- доступ к GitHub, GitLab или Bitbucket;
- корректно настроенные `user.name` и `user.email` в Git;
- подтверждённый email на вашей Git-платформе, совпадающий с email автора коммита.

Проверьте версию Git:

```bash
git --version
```

Проверьте текущую Git-идентичность:

```bash
git config --global user.name
git config --global user.email
```

Проверьте существующие SSH-ключи:

```bash
ls -al ~/.ssh
```

## Архитектура настройки

Есть два распространённых подхода:

### Вариант A — использовать текущий SSH-ключ
Это быстрее.

Подходит для:
- личных проектов;
- быстрой настройки;
- одиночной работы.

### Вариант B — создать отдельный SSH-ключ только для подписи
Это чище и удобнее в сопровождении.

Подходит для:
- production- или рабочих окружений;
- разделения ролей и обязанностей;
- более простой ротации и аудита;
- более чистых демо в обучающем видео.

**Рекомендуется для курса:** использовать отдельный ключ для подписи.

## Шаг 1. Создайте отдельный SSH-ключ для подписи

Используйте **Ed25519**, если только у вашей организации нет отдельного требования на другой алгоритм.

```bash
ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/id_ed25519_signing
```

Что делает эта команда:

- создаёт приватный ключ: `~/.ssh/id_ed25519_signing`
- создаёт публичный ключ: `~/.ssh/id_ed25519_signing.pub`
- добавляет к ключу комментарий, обычно ваш email

Когда появится запрос:
- выберите надёжную парольную фразу;
- не оставляйте приватный ключ без защиты, если полностью не понимаете риск.

Запустите `ssh-agent` и добавьте ключ:

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519_signing
```

Покажите публичный ключ:

```bash
cat ~/.ssh/id_ed25519_signing.pub
```

На macOS скопируйте его в буфер обмена:

```bash
pbcopy < ~/.ssh/id_ed25519_signing.pub
```

---

## Шаг 2. Добавьте ключ на вашу Git-платформу

### GitHub

В GitHub перейдите в:

**Settings → SSH and GPG keys → New SSH key**

Затем:

1. введите название, например `MacBook signing key`;
2. выберите тип ключа **Signing**;
3. вставьте публичный ключ;
4. сохраните его.

Важная деталь:

Если вы хотите использовать **один и тот же SSH-ключ** и для Git-аутентификации, и для подписи коммитов в GitHub, его нужно загрузить **дважды**: один раз как ключ аутентификации и один раз как ключ подписи.

Альтернатива через CLI:

```bash
gh ssh-key add ~/.ssh/id_ed25519_signing.pub --type signing --title "MacBook signing key"
```

### GitLab

В GitLab добавьте ключ с типом использования:

- **Signing**, или
- **Authentication & Signing**

Также убедитесь, что email в коммите совпадает с **подтверждённым email** в вашем аккаунте GitLab.

### Bitbucket

Bitbucket тоже поддерживает подпись коммитов через SSH, но подтверждение SSH-подписанных коммитов в основном ориентировано на коммиты, отправленные из CLI.

---

## Шаг 3. Укажите Git использовать SSH для подписи

Настройте Git глобально:

```bash
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519_signing.pub
```

Что означают эти параметры:

- `gpg.format ssh` указывает Git использовать SSH-подписи вместо OpenPGP/GPG;
- `user.signingkey` указывает Git на файл **публичного** SSH-ключа, который определяет, какой ключ использовать для подписи.

Необязательно, но настоятельно рекомендуется:

```bash
git config --global commit.gpgsign true
git config --global tag.gpgSign true
```

Это заставит Git подписывать:

- каждый коммит по умолчанию;
- каждый тег по умолчанию.

Вы можете проверить конфигурацию:

```bash
git config --global --list | grep -E 'gpg.format|user.signingkey|commit.gpgsign|tag.gpgSign'
```

## Шаг 4. Создайте первый подписанный коммит

Внутри любого репозитория:

```bash
git checkout -b demo/ssh-signing
echo "SSH signing demo" > signing-demo.txt
git add signing-demo.txt
git commit -S -m "feat: add SSH signing demo"
```

Если `commit.gpgsign=true` уже установлен, можно просто выполнить:

```bash
git commit -m "feat: add SSH signing demo"
```

Отправьте ветку:

```bash
git push -u origin demo/ssh-signing
```

Ожидаемый результат:

- Git запросит парольную фразу для ключа, если это нужно;
- коммит будет отправлен как обычно;
- в UI платформы должна появиться метка **Verified**, как только платформа сможет проверить подпись.

---

## Шаг 5. Проверьте результат в UI платформы

### На GitHub
Корректный SSH-подписанный коммит может отображаться как **Verified** или, в некоторых случаях, как **Partially verified** в зависимости от условий репозитория и vigilant mode.

### На GitLab
Корректный SSH-подписанный коммит отображается как **Verified**.

### На Bitbucket
SSH-подписанные коммиты также могут быть проверены, но документированный сценарий сосредоточен на коммитах, отправленных через CLI.

Если подпись есть, но платформа не может корректно доверять ей или связать её с аккаунтом, коммит может отображаться как **Unverified**.

---

## Шаг 6. Проверяйте подписи локально

Проверка на платформе полезна, но локальная проверка — важная часть понимания того, как работает SSH-подпись.

Для SSH-подписей локальная проверка зависит от **файла разрешённых подписантов**.

Создайте его:

```bash
mkdir -p ~/.ssh
touch ~/.ssh/allowed_signers
git config --global gpg.ssh.allowedSignersFile ~/.ssh/allowed_signers
```

Добавьте в файл ваш публичный ключ:

```bash
echo "$(git config --get user.email) namespaces=\"git\" $(cat ~/.ssh/id_ed25519_signing.pub)" >> ~/.ssh/allowed_signers
```

Теперь проверьте подписи:

```bash
git log --show-signature -1
```

Вы должны увидеть примерно такой вывод:

```text
Good "git" signature for your_email@example.com with ED25519 key ...
```

### Почему важен `allowed_signers`

SSH-подпись работает не так, как модель web-of-trust в GPG. Git нужен файл, который явно указывает, каким публичным ключам вы доверяете при проверке. Для этого и нужен `gpg.ssh.allowedSignersFile`.

---

## Шаг 7. Подписывайте и теги тоже

Если вы хотите подписанные релизы:

```bash
git tag -s v0.1.0 -m "v0.1.0"
git push origin v0.1.0
```

Если включён `tag.gpgSign=true`, теги тоже будут подписываться по умолчанию.

Это особенно полезно для:

- тегов релизов;
- публичных репозиториев;
- внутренних пайплайнов доставки;
- подтверждения происхождения точек релиза.

## Частые причины, почему не появляется `Verified`

### 1. Git слишком старый
Для проверки SSH-подписей нужен **Git 2.34+**.

### 2. Ключ не был добавлен как ключ подписи
Это часто случается на GitHub, если ключ был загружен только для аутентификации.

### 3. Неверный email автора коммита
Email в коммите должен совпадать с подтверждённым email в аккаунте платформы.

Проверьте это:

```bash
git config user.email
git config --global user.email
```

### 4. Вы указали неверный путь к ключу
`user.signingkey` должен указывать на правильный `.pub`-файл.

### 5. В `ssh-agent` не загружен приватный ключ
Исправление:

```bash
ssh-add ~/.ssh/id_ed25519_signing
```

### 6. Локальная проверка не работает, потому что отсутствует `allowed_signers`
Проверка в UI и локальная проверка связаны, но не идентичны. Платформа может показывать **Verified**, даже если локальная машина ещё не умеет проверять подписи, потому что локальному Git всё ещё нужен настроенный `gpg.ssh.allowedSignersFile`.

### 7. Вы переписали историю и проверяете устаревшие предположения
Всегда проверяйте точный SHA коммита после rebase, squash или amended-коммитов.

## Сценарий демо для видео

Используйте в скринкасте такую последовательность:

1. покажите неподписанный коммит в истории репозитория;
2. за 20–30 секунд объясните проблему доверия;
3. создайте отдельный ключ для подписи;
4. добавьте ключ в GitHub как **Signing**;
5. настройте Git с `gpg.format ssh` и `user.signingkey`;
6. создайте тестовый коммит;
7. отправьте его;
8. обновите GitHub и покажите метку **Verified**;
9. проверьте подпись локально через `git log --show-signature`;
10. один раз намеренно сломайте настройку, например через неверный email или отсутствующий ключ подписи, и покажите режим отказа.

Так видео будет выглядеть убедительнее, потому что зрители увидят и успешный сценарий, и типичную ошибку.

## Минимальная шпаргалка по командам

```bash
# 1) Создать ключ для подписи
ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/id_ed25519_signing

# 2) Добавить в ssh-agent
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519_signing

# 3) Настроить Git для SSH-подписи
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519_signing.pub
git config --global commit.gpgsign true
git config --global tag.gpgSign true

# 4) Создать подписанный коммит
git add .
git commit -S -m "feat: signed commit with SSH"

# 5) Локальная проверка
mkdir -p ~/.ssh
touch ~/.ssh/allowed_signers
git config --global gpg.ssh.allowedSignersFile ~/.ssh/allowed_signers
echo "$(git config --get user.email) namespaces=\"git\" $(cat ~/.ssh/id_ed25519_signing.pub)" >> ~/.ssh/allowed_signers
git log --show-signature -1
```
