# Підпис Git-комітів за допомогою SSH-ключів

> Міні-курс і практичне керівництво для YouTube-відео та GitHub-репозиторію.

---

## Опис курсу

Цей міні-курс пояснює, як підписувати Git-коміти та теги SSH-ключем, як налаштувати автоматичний підпис комітів у Git і як перевіряти ці підписи локально та на платформах на кшталт GitHub, GitLab і Bitbucket.

На відміну від класичного сценарію з GPG, підпис через SSH зазвичай простіше налаштувати, адже багато розробників вже використовують SSH-ключі для доступу до Git. GitHub прямо описує SSH як найпростіший варіант для більшості індивідуальних користувачів, а для перевірки SSH-підписів Git вимагає версію **2.34 або новішу**. GitHub, GitLab і Bitbucket можуть показувати мітку **Verified** для коректно підписаних комітів через SSH, якщо ключ підпису прив'язаний до вашого акаунту, а метадані коміту відповідають вимогам платформи.

---

## Чому це варто робити

Підпис комітів вирішує іншу задачу, ніж просто аутентифікація при роботі з Git по SSH.

Аутентифікація відповідає на запитання:

**«Чи може ця машина надсилати зміни до репозиторію?»**

Підпис коміту відповідає на інше запитання:

**«Чи можу я криптографічно довести, хто створив цей коміт?»**

Це важливо, тому що підпис комітів:

- підвищує довіру до історії змін;
- допомагає рецензентам підтверджувати походження комітів;
- робить ваші зміни позначеними як **Verified** на платформах для хостингу репозиторіїв;
- знижує неоднозначність у командній роботі, автоматизації та open-source-співпраці;
- ускладнює підміну історії та видавання себе за іншого автора;
- допомагає організаціям вимагати підписані коміти у захищених гілках.

Для аудиторії YouTube головний посил такий:

> SSH-підпис — це один із найпростіших «маленьких апгрейдів безпеки», який можна додати до щоденного Git-процесу.

---

## Що студент отримає в кінці

Після цього керівництва студент зможе:

1. розуміти, що таке підпис Git-комітів через SSH і чим він відрізняється від звичайної SSH-аутентифікації;
2. створити окремий SSH-ключ для підпису або повторно використати наявний;
3. зареєструвати публічний ключ у GitHub, GitLab або Bitbucket;
4. налаштувати Git на підпис комітів і тегів через SSH;
5. підписувати коміти вручну та автоматично;
6. перевіряти підписи локально за допомогою `git log --show-signature`;
7. розуміти, чому коміт може відображатися як `Verified`, `Unverified` або взагалі без статусу.

- Сценарій для GitHub.

## Попередні вимоги

Перед початком переконайтеся, що у вас є:

- **Git 2.34+** для перевірки SSH-підписів;
- SSH-ключ або право його створити;
- доступ до GitHub, GitLab або Bitbucket;
- коректно налаштовані `user.name` і `user.email` у Git;
- підтверджений email на вашій Git-платформі, що збігається з email автора коміту.

Перевірте версію Git:

```bash
git --version
```

Перевірте поточну Git-ідентичність:

```bash
git config --global user.name
git config --global user.email
```

Перевірте наявні SSH-ключі:

```bash
ls -al ~/.ssh
```

## Архітектура налаштування

Є два поширені підходи:

### Варіант A — використати поточний SSH-ключ
Це швидше.

Підходить для:
- особистих проектів;
- швидкого налаштування;
- одиночної роботи.

### Варіант B — створити окремий SSH-ключ тільки для підпису
Це чистіше і зручніше в обслуговуванні.

Підходить для:
- production- або робочих середовищ;
- розділення ролей і обов'язків;
- простішої ротації та аудиту;
- чистіших демо у навчальному відео.

**Рекомендується для курсу:** використовувати окремий ключ для підпису.

## Крок 1. Створіть окремий SSH-ключ для підпису

Використовуйте **Ed25519**, якщо тільки у вашої організації немає окремої вимоги до іншого алгоритму.

```bash
ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/id_ed25519_signing
```

Що робить ця команда:

- створює приватний ключ: `~/.ssh/id_ed25519_signing`
- створює публічний ключ: `~/.ssh/id_ed25519_signing.pub`
- додає до ключа коментар, зазвичай ваш email

Коли з'явиться запит:
- оберіть надійну парольну фразу;
- не залишайте приватний ключ без захисту, якщо повністю не розумієте ризик.

Запустіть `ssh-agent` і додайте ключ:

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519_signing
```

Покажіть публічний ключ:

```bash
cat ~/.ssh/id_ed25519_signing.pub
```

На macOS скопіюйте його до буфера обміну:

```bash
pbcopy < ~/.ssh/id_ed25519_signing.pub
```

---

## Крок 2. Додайте ключ на вашу Git-платформу

### GitHub

У GitHub перейдіть до:

**Settings → SSH and GPG keys → New SSH key**

Потім:

1. введіть назву, наприклад `MacBook signing key`;
2. оберіть тип ключа **Signing**;
3. вставте публічний ключ;
4. збережіть його.

Важлива деталь:

Якщо ви хочете використовувати **один і той самий SSH-ключ** і для Git-аутентифікації, і для підпису комітів у GitHub, його потрібно завантажити **двічі**: один раз як ключ аутентифікації і один раз як ключ підпису.

Альтернатива через CLI:

```bash
gh ssh-key add ~/.ssh/id_ed25519_signing.pub --type signing --title "MacBook signing key"
```

### GitLab

У GitLab додайте ключ з типом використання:

- **Signing**, або
- **Authentication & Signing**

Також переконайтеся, що email у коміті збігається з **підтвердженим email** у вашому акаунті GitLab.

### Bitbucket

Bitbucket також підтримує підпис комітів через SSH, але підтвердження SSH-підписаних комітів в основному орієнтоване на коміти, надіслані з CLI.

---

## Крок 3. Вкажіть Git використовувати SSH для підпису

Налаштуйте Git глобально:

```bash
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519_signing.pub
```

Що означають ці параметри:

- `gpg.format ssh` вказує Git використовувати SSH-підписи замість OpenPGP/GPG;
- `user.signingkey` вказує Git на файл **публічного** SSH-ключа, який визначає, який ключ використовувати для підпису.

Необов'язково, але наполегливо рекомендується:

```bash
git config --global commit.gpgsign true
git config --global tag.gpgSign true
```

Це змусить Git підписувати:

- кожен коміт за замовчуванням;
- кожен тег за замовчуванням.

Ви можете перевірити конфігурацію:

```bash
git config --global --list | grep -E 'gpg.format|user.signingkey|commit.gpgsign|tag.gpgSign'
```

---

## Крок 4. Створіть перший підписаний коміт

Всередині будь-якого репозиторію:

```bash
git checkout -b demo/ssh-signing
echo "SSH signing demo" > signing-demo.txt
git add signing-demo.txt
git commit -S -m "feat: add SSH signing demo"
```

Якщо `commit.gpgsign=true` вже встановлено, можна просто виконати:

```bash
git commit -m "feat: add SSH signing demo"
```

Надішліть гілку:

```bash
git push -u origin demo/ssh-signing
```

Очікуваний результат:

- Git запросить парольну фразу для ключа, якщо це потрібно;
- коміт буде надіслано як зазвичай;
- в UI платформи має з'явитися мітка **Verified**, як тільки платформа зможе перевірити підпис.

---

## Крок 5. Перевірте результат в UI платформи

### На GitHub
Коректний SSH-підписаний коміт може відображатися як **Verified** або, у деяких випадках, як **Partially verified** залежно від умов репозиторію та vigilant mode.

### На GitLab
Коректний SSH-підписаний коміт відображається як **Verified**.

### На Bitbucket
SSH-підписані коміти також можуть бути перевірені, але задокументований сценарій зосереджений на комітах, надісланих через CLI.

Якщо підпис є, але платформа не може коректно йому довіряти або пов'язати його з акаунтом, коміт може відображатися як **Unverified**.

---

## Крок 6. Перевіряйте підписи локально

Перевірка на платформі корисна, але локальна перевірка — важлива частина розуміння того, як працює SSH-підпис.

Для SSH-підписів локальна перевірка залежить від **файлу дозволених підписувачів**.

Створіть його:

```bash
mkdir -p ~/.ssh
touch ~/.ssh/allowed_signers
git config --global gpg.ssh.allowedSignersFile ~/.ssh/allowed_signers
```

Додайте до файлу ваш публічний ключ:

```bash
echo "$(git config --get user.email) namespaces=\"git\" $(cat ~/.ssh/id_ed25519_signing.pub)" >> ~/.ssh/allowed_signers
```

Тепер перевірте підписи:

```bash
git log --show-signature -1
```

Ви маєте побачити приблизно такий вивід:

```text
Good "git" signature for your_email@example.com with ED25519 key ...
```

### Чому важливий `allowed_signers`

SSH-підпис працює не так, як модель web-of-trust у GPG. Git потребує файл, який явно вказує, яким публічним ключам ви довіряєте при перевірці. Для цього і потрібен `gpg.ssh.allowedSignersFile`.

---

## Крок 7. Підписуйте і теги теж

Якщо ви хочете підписані релізи:

```bash
git tag -s v0.1.0 -m "v0.1.0"
git push origin v0.1.0
```

Якщо увімкнено `tag.gpgSign=true`, теги також підписуватимуться за замовчуванням.

Це особливо корисно для:

- тегів релізів;
- публічних репозиторіїв;
- внутрішніх пайплайнів доставки;
- підтвердження походження точок релізу.

---

## Часті причини, чому не з'являється `Verified`

### 1. Git занадто старий
Для перевірки SSH-підписів потрібен **Git 2.34+**.

### 2. Ключ не був доданий як ключ підпису
Це часто трапляється на GitHub, якщо ключ був завантажений тільки для аутентифікації.

### 3. Невірний email автора коміту
Email у коміті має збігатися з підтвердженим email в акаунті платформи.

Перевірте це:

```bash
git config user.email
git config --global user.email
```

### 4. Ви вказали невірний шлях до ключа
`user.signingkey` має вказувати на правильний `.pub`-файл.

### 5. В `ssh-agent` не завантажено приватний ключ
Виправлення:

```bash
ssh-add ~/.ssh/id_ed25519_signing
```

### 6. Локальна перевірка не працює, бо відсутній `allowed_signers`
Перевірка в UI та локальна перевірка пов'язані, але не ідентичні. Платформа може показувати **Verified**, навіть якщо локальна машина ще не вміє перевіряти підписи, бо локальному Git все ще потрібен налаштований `gpg.ssh.allowedSignersFile`.

### 7. Ви переписали історію і перевіряєте застарілі припущення
Завжди перевіряйте точний SHA коміту після rebase, squash або amended-комітів.

## Пропоновані тези для пояснення на YouTube

### Основний посил
«SSH-підпис — це не про доступ до репозиторію. Це про доведення авторства коміту».

### Сильне порівняльне формулювання
«SSH-аутентифікація доводить, що ви можете надсилати зміни. SSH-підпис доводить, що коміт прийшов від довіреного ключа».

### Практичний аспект безпеки
«Це одне з рідкісних покращень безпеки, яке одночасно реально корисне і дешеве у впровадженні: кілька команд, і історія стає надійнішою».

### Чистий навчальний ракурс
«Якщо ви вже використовуєте SSH для Git, то підпис комітів через SSH зазвичай простіший, ніж класичний шлях через GPG».

## Сценарій демо для відео

Використовуйте у скринкасті таку послідовність:

1. покажіть непідписаний коміт в історії репозиторію;
2. за 20–30 секунд поясніть проблему довіри;
3. створіть окремий ключ для підпису;
4. додайте ключ у GitHub як **Signing**;
5. налаштуйте Git з `gpg.format ssh` і `user.signingkey`;
6. створіть тестовий коміт;
7. надішліть його;
8. оновіть GitHub і покажіть мітку **Verified**;
9. перевірте підпис локально через `git log --show-signature`;
10. один раз навмисно зламайте налаштування, наприклад через невірний email або відсутній ключ підпису, і покажіть режим відмови.

Так відео виглядатиме переконливіше, бо глядачі побачать і успішний сценарій, і типову помилку.

## Мінімальна шпаргалка по командах

```bash
# 1) Створити ключ для підпису
ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/id_ed25519_signing

# 2) Додати до ssh-agent
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519_signing

# 3) Налаштувати Git для SSH-підпису
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519_signing.pub
git config --global commit.gpgsign true
git config --global tag.gpgSign true

# 4) Створити підписаний коміт
git add .
git commit -S -m "feat: signed commit with SSH"

# 5) Локальна перевірка
mkdir -p ~/.ssh
touch ~/.ssh/allowed_signers
git config --global gpg.ssh.allowedSignersFile ~/.ssh/allowed_signers
echo "$(git config --get user.email) namespaces=\"git\" $(cat ~/.ssh/id_ed25519_signing.pub)" >> ~/.ssh/allowed_signers
git log --show-signature -1
```
